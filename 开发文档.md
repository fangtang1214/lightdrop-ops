# 轻放直播电商运营助手 开发文档

## 1. 项目英文名

**LightDrop LiveOps Assistant**

建议项目根目录命名：

```text
lightdrop-liveops-assistant
```

备选简短命名：

```text
lightdrop-ops
```

推荐使用 **lightdrop-liveops-assistant**，含义清晰，适合长期扩展。

---

## 2. 项目定位

**轻放直播电商运营助手** 是一款面向直播电商运营人员的电脑端数据分析工具，主要用于对直播后表现较差的商品进行图片筛查、重复识别、统计分析和报告生成。

用户每场直播结束后，会将表现不好的商品图片保存到独立文件夹中。程序需要读取这些文件夹中的图片，对不同直播场次中的商品图片进行相似度匹配，找出多次出现在“差品列表”中的商品，并生成可视化数据报告。

## 3. 核心使用场景

### 3.1 用户当前工作方式

用户每场直播结束后，会手动保存表现不好的商品图片。

示例文件夹结构：

```text
差品图片库/
├── 2026-05-01_第1场/
│   ├── 商品A.jpg
│   ├── 商品B.jpg
│   └── 商品C.jpg
├── 2026-05-03_第2场/
│   ├── 商品A_截图.png
│   ├── 商品D.jpg
│   └── 商品E.jpg
├── 2026-05-05_第3场/
│   ├── 商品B.jpg
│   ├── 商品F.jpg
│   └── 商品G.jpg
└── 2026-05-07_第4场/
    ├── 商品A.jpg
    ├── 商品B_新图.png
    └── 商品H.jpg
```

### 3.2 程序需要解决的问题

程序需要判断：

* 哪些商品在最近 4 场直播中多次成为差品
* 哪些商品在最近 6 场直播中多次成为差品
* 哪些商品在最近 10 场直播中多次成为差品
* 哪些商品长期重复出现，需要重点处理
* 哪些商品只偶尔出现，优先级较低
* 哪些场次的差品重复率较高

## 4. 推荐技术方案

### 4.1 前端形式建议

建议使用 **网页前端 + 本地桌面程序壳** 的方式。

也就是说：

* 界面使用网页技术开发
* 程序以电脑软件形式运行
* 用户不需要自己打开浏览器部署服务
* 数据和图片都保存在本地电脑

推荐方案：

```text
Electron + React + Python 后端服务
```

或者：

```text
Tauri + React + Python 后端服务
```

### 4.2 为什么建议使用网页前端

网页前端更适合这个项目，因为数据报告需要较丰富的展示形式，例如：

* 商品图片卡片
* 多场次对比表格
* 最近 4 / 6 / 10 场筛选标签
* 重复出现次数排行榜
* 商品相似度匹配结果
* 图表统计
* 可点击查看商品出现在哪几场直播
* 可导出 Excel / PDF 报告

如果使用传统桌面 GUI，界面开发会更麻烦，视觉效果也不如网页灵活。

### 4.3 推荐开发栈

#### 前端

```text
React + TypeScript + Vite
```

UI 组件库建议：

```text
Ant Design 或 shadcn/ui
```

图表库建议：

```text
ECharts 或 Recharts
```

#### 桌面端壳

优先推荐：

```text
Electron
```

原因：

* 生态成熟
* 桌面程序开发资料多
* 适合 Windows 环境
* 和网页前端结合简单

#### 后端

```text
Python + FastAPI
```

原因：

* Python 适合图片处理和 AI 图像匹配
* FastAPI 开发本地接口方便
* 后续可接入更高级的图像识别模型

#### 图片处理

基础阶段：

```text
OpenCV + imagehash
```

进阶阶段：

```text
CLIP 图像向量模型
```

### 4.4 推荐总体架构

```text
Electron 桌面程序
└── React 前端界面
    └── 调用本地 FastAPI 服务
        ├── 扫描文件夹
        ├── 读取图片
        ├── 计算图片指纹 / 图片向量
        ├── 执行相似度匹配
        ├── 生成统计数据
        └── 返回报告结果
```

## 5.3 图片匹配模块

### 功能说明

判断不同场次中的图片是否属于同一个商品。

因为同一个商品可能会出现以下情况：

* 文件名不同
* 图片裁剪不同
* 清晰度不同
* 角度不同
* 水印不同
* 背景略有不同

所以不能只依赖文件名，需要根据图片内容进行匹配。

### 第一阶段匹配方式

第一阶段可以使用图片哈希算法：

```text
aHash / pHash / dHash
```

优点：

* 开发简单
* 速度快
* 适合相同或高度相似的图片

缺点：

* 如果商品角度、背景、裁剪变化较大，识别效果会下降

### 第二阶段匹配方式

第二阶段建议使用 CLIP 图像向量模型。

优点：

* 可以识别更复杂的相似商品图片
* 不要求图片完全一样
* 对不同角度、不同截图、不同背景更友好

缺点：

* 开发复杂度更高
* 首次计算会慢一些
* 需要本地模型文件或联网模型服务

### 推荐实现策略

建议采用两级匹配：

```text
第一层：图片哈希快速匹配
第二层：CLIP 向量相似度匹配
```

匹配流程：

```text
读取图片
→ 生成图片 hash
→ 生成图片向量
→ 与历史图片进行相似度比较
→ 超过阈值则归为同一商品
→ 生成商品分组
```

## 5.4 商品分组模块

### 功能说明

程序需要把被判断为同一商品的图片归到一个商品组里。

### 商品组数据结构

```json
{
  "product_group_id": "PG_000001",
  "representative_image": "D:/xxx/商品A.jpg",
  "images": [
    "D:/xxx/2026-05-01_第1场/商品A.jpg",
    "D:/xxx/2026-05-03_第2场/商品A_截图.png",
    "D:/xxx/2026-05-07_第4场/商品A.jpg"
  ],
  "appeared_sessions": [
    "2026-05-01_第1场",
    "2026-05-03_第2场",
    "2026-05-07_第4场"
  ],
  "appear_count": 3
}
```

### 重要规则

同一个商品在同一场直播文件夹里即使有多张图片，也应该只算作该场出现 1 次。

例如：

```text
第1场里有 商品A_1.jpg、商品A_2.jpg、商品A_3.jpg
```

统计时应该认为：

```text
商品A 在第1场出现 1 次
```

## 5.5 数据统计模块

### 核心统计维度

程序需要支持以下统计：

* 最近 4 场出现的差品
* 最近 6 场出现的差品
* 最近 10 场出现的差品
* 全部场次高频差品
* 单个商品出现次数
* 单个商品出现过哪些场次
* 每场直播差品数量
* 每场直播重复差品数量
* 每场直播新增差品数量
* 差品重复率

### 最近 N 场统计逻辑

用户选择最近 N 场后，程序读取排序后的最后 N 个场次。

示例：

```text
共有 12 场直播
最近 4 场 = 第9场、第10场、第11场、第12场
最近 6 场 = 第7场 至 第12场
最近 10 场 = 第3场 至 第12场
```

### 最近 N 场出现差品定义

在最近 N 场中，出现次数大于等于 2 的商品，默认认为是“重复差品”。

可以提供筛选条件：

```text
出现次数 >= 2
出现次数 >= 3
出现次数 >= 4
```

## 7. 数据库设计

建议使用本地数据库：

```text
SQLite
```

### 7.1 sessions 表

用于保存直播场次。

```sql
CREATE TABLE sessions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    folder_path TEXT NOT NULL,
    session_index INTEGER NOT NULL,
    created_at TEXT,
    updated_at TEXT
);
```

### 7.2 images 表

用于保存图片信息。

```sql
CREATE TABLE images (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id INTEGER NOT NULL,
    file_name TEXT NOT NULL,
    file_path TEXT NOT NULL,
    width INTEGER,
    height INTEGER,
    file_size INTEGER,
    image_hash TEXT,
    vector_path TEXT,
    product_group_id INTEGER,
    created_at TEXT,
    FOREIGN KEY(session_id) REFERENCES sessions(id)
);
```

### 7.3 product_groups 表

用于保存商品分组。

```sql
CREATE TABLE product_groups (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    representative_image_path TEXT,
    created_at TEXT,
    updated_at TEXT
);
```

### 7.4 product_group_sessions 表

用于保存商品组和直播场次的关系。

```sql
CREATE TABLE product_group_sessions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    product_group_id INTEGER NOT NULL,
    session_id INTEGER NOT NULL,
    appear_count INTEGER DEFAULT 1,
    FOREIGN KEY(product_group_id) REFERENCES product_groups(id),
    FOREIGN KEY(session_id) REFERENCES sessions(id)
);
```

## 9. 匹配参数建议

### 9.1 哈希匹配阈值

```text
汉明距离 <= 8：高度相似
汉明距离 <= 12：可能相似
汉明距离 > 12：大概率不同
```

### 9.2 CLIP 相似度阈值

```text
相似度 >= 0.90：高度相似
相似度 >= 0.85：可能相似
相似度 < 0.85：大概率不同
```

### 9.3 人工确认机制

建议提供“疑似相似商品”人工确认功能。

当相似度处于中间区间时，例如：

```text
0.80 - 0.85
```

程序不直接合并，而是进入人工确认列表。

用户可以选择：

```text
确认同一商品
确认不是同一商品
忽略
```

## 14. 程序核心价值

这个程序的核心价值不是简单管理图片，而是帮助直播电商运营人员快速发现：

```text
哪些商品反复表现不好
哪些商品应该暂停投放
哪些商品需要重新拍图或优化话术
哪些商品只是偶发性表现差
```
